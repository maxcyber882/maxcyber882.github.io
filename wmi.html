Core components that make WMI work

What is WMI? The Simple Explanation

Windows Management Instrumentation (WMI) is like a massive built-in database that knows everything about your Windows computer. It can tell you what hardware you have, what software is installed, what processes are running, who is logged in, and much more.

Think of WMI as a librarian who knows everything about your computer and can answer any question you ask. You just need to know how to ask the questions.

The Struggle I Faced: When I first tried to understand WMI, I was overwhelmed by complex documentation and technical jargon. But after months of practice, I realized WMI is simply a way to ask Windows questions and get answers. This article is my attempt to explain it the way I wish someone had explained it to me.

WMI is important because:

  • It's built into every Windows computer since Windows 2000
  • It can retrieve any system information you can think of
  • It can execute commands on remote computers
  • It can monitor system events in real-time
  • It's used by system administrators and attackers alike

How WMI Actually Works

At its core, WMI works through a simple question-and-answer system. You ask a question using a specific format, and WMI gives you the answer.

The WMI Database (Repository)

WMI stores information in a structured database called the WMI Repository. This database is stored in the C:\Windows\System32\wbem\Repository folder. When you ask a question, WMI looks up the answer in this database.

// How Information Gets into WMI 1. Windows components and applications register with WMI 2. They tell WMI what information they can provide 3. WMI providers collect this information 4. Information is stored in the WMI Repository 5. When you query WMI, it reads from this repository

WMI Providers: The Data Collectors

WMI doesn't know everything automatically. It relies on providers - small programs that know how to get specific types of information. For example:

// Examples of WMI Providers - Registry Provider: Reads Windows Registry information - Win32 Provider: Gets hardware and OS information - Security Provider: Handles security-related data - Performance Counter Provider: Gets performance metrics - Event Log Provider: Reads Windows event logs

When you query for information, WMI asks the appropriate provider to fetch the data for you. You don't need to know which provider to use - WMI handles that automatically.

The WMI Namespace Structure

WMI organizes information in a hierarchical structure called namespaces. Think of it like folders on your computer.

// WMI Namespace Structure root ├── root\default │ └── Basic system information ├── root\cimv2 │ └── Most common WMI classes (where 90% of queries happen) ├── root\Security │ └── Security-related information ├── root\Microsoft │ └── Microsoft-specific extensions ├── root\RSOP │ └── Resultant Set of Policy information └── root\Policy └── Group Policy information
The Most Important Namespace: root\cimv2 is where you'll spend most of your time. It contains thousands of classes covering everything from BIOS information to running processes.

WMI Classes: The Building Blocks

In WMI, everything is organized into classes. A class is like a template that defines what information can be collected about a particular thing.

Understanding WMI Classes

Each class has properties (the data it stores) and methods (actions it can perform). For example, the Win32_Process class:

// Win32_Process Class Example Class: Win32_Process Properties: - Name: Name of the process (e.g., "notepad.exe") - ProcessId: Unique number identifying the process - WorkingSetSize: How much memory the process uses - CreationDate: When the process started Methods: - Create: Start a new process - Terminate: Stop a running process - GetOwner: Find who owns the process

Common WMI Classes You'll Use

Class Name What It Tells You Example Query
Win32_ComputerSystem Computer name, manufacturer, total RAM Get total memory installed
Win32_OperatingSystem Windows version, last boot time, OS architecture Check if system is 64-bit
Win32_Process Running processes, their IDs, memory usage List all running programs
Win32_Service Windows services, their status, start mode Check if firewall is running
Win32_LogicalDisk Disk drives, free space, total size Check available disk space
Win32_NetworkAdapterConfiguration IP addresses, MAC addresses, DNS settings Get computer's IP address
Win32_Product Installed software and applications List installed programs

How to Query WMI (The Simple Way)

There are several ways to ask WMI questions. Let's start with the simplest.

Method 1: WMIC (Command Line)

WMIC (WMI Command-line) is the easiest way to get started. Open Command Prompt and try these commands:

// Basic WMIC Commands wmic os get caption This shows your Windows version wmic computersystem get totalphysicalmemory This shows total RAM in bytes wmic process list brief This shows all running processes wmic service where "name like 'wmi%'" get name,state This shows WMI-related services wmic logicaldisk where "drivetype=3" get deviceid,size,freespace This shows all hard drives and free space

Method 2: PowerShell (More Powerful)

PowerShell provides a much more flexible way to work with WMI:

// PowerShell WMI Commands Get-WmiObject -Class Win32_ComputerSystem This returns computer information Get-WmiObject -Class Win32_Process | Select-Object Name,ProcessId This lists all processes with names and IDs Get-WmiObject -Class Win32_Service | Where-Object {$_.State -eq "Running"} This shows only running services Get-WmiObject -Class Win32_LogicalDisk -Filter "DriveType=3" | Select-Object DeviceId,@{Name="FreeGB";Expression={$_.FreeSpace/1GB}} This shows free space on all drives in GB

Method 3: WQL (WMI Query Language)

WQL is like SQL but for WMI. It allows you to write precise queries:

// WQL Query Examples SELECT * FROM Win32_Process WHERE Name = 'notepad.exe' Find all Notepad processes SELECT Name,ProcessId,WorkingSetSize FROM Win32_Process WHERE WorkingSetSize > 50000000 Find processes using more than 50MB memory SELECT * FROM Win32_Service WHERE State = 'Stopped' AND StartMode = 'Auto' Find services that should be running but aren't

The Power of Remote WMI

One of WMI's most powerful features is the ability to query remote computers on your network.

// Remote WMI with PowerShell Get-WmiObject -Class Win32_ComputerSystem -ComputerName "DESKTOP-ABC123" Get-WmiObject -Class Win32_Process -ComputerName "SERVER01" | Select-Object Name,ProcessId Get-WmiObject -Class Win32_Service -ComputerName "LAPTOP-XYZ" -Filter "State='Running'" // Remote WMI with WMIC wmic /node:"SERVER01" os get caption wmic /node:"DESKTOP-ABC123" logicaldisk where "drivetype=3" get deviceid,freespace
Prerequisites for Remote WMI: - The RemoteRegistry service must be running on target - Windows Firewall must allow WMI traffic (port 135) - You need appropriate permissions on the target - Your account must have administrative rights

WMI Events: When Things Happen

WMI can not only answer questions but also notify you when something changes. This is called WMI Eventing.

Monitor Process Creation

// PowerShell script to watch for new processes Register-WmiEvent -Query "SELECT * FROM Win32_ProcessStartTrace" -Action { Write-Host "Process started: $($event.SourceEventArgs.NewEvent.ProcessName)" }

Monitor When Someone Logs In

Register-WmiEvent -Query "SELECT * FROM Win32_LogonSession" -Action { Write-Host "User logged in" }

Monitor Disk Space

$query = "SELECT * FROM __InstanceModificationEvent WITHIN 60 WHERE TargetInstance ISA 'Win32_LogicalDisk' AND TargetInstance.FreeSpace < 1073741824" Register-WmiEvent -Query $query -Action { Write-Host "WARNING: Low disk space detected!" }

Why Attackers Love WMI

WMI is powerful, and attackers have learned to use it too. Here's why:

// Malicious Uses of WMI 1. Execution: Run commands on remote systems wmic /node:"target" process call create "malware.exe" 2. Persistence: Run malware at startup wmic /namespace:"\\root\subscription" path __EventFilter create ... 3. Lateral Movement: Move between computers Invoke-WmiMethod -Class Win32_Process -Name Create -ArgumentList "payload.exe" -ComputerName "target" 4. Information Gathering: Collect system data Get-WmiObject -Class Win32_UserAccount Get-WmiObject -Class Win32_Product
Security Note: Because WMI is so powerful and legitimate, it's difficult to distinguish between administrative use and malicious use. This is why WMI is often called "living off the land" for attackers.

// Check these: 1. Windows Firewall: Allow "Windows Management Instrumentation (WMI)" 2. DCOM permissions: Check security settings 3. User permissions: Must be admin on remote machine 4. Services: Remote Registry and WMI services must be running

The Mental Model That Finally Made WMI Click For Me

After struggling with WMI for months, this is the mental model that finally made everything make sense:

// My Mental Model for WMI WMI is like a massive Excel spreadsheet that Windows maintains about itself. Think of it this way: - Classes are like worksheet tabs (Processes, Services, Disks) - Properties are like columns (Name, ID, Memory) - Instances are like rows (Notepad.exe, Explorer.exe) Every query is just asking: "Go to this worksheet, filter these rows, show me these columns" That's it. That's all WMI does. It's just a database of your system information.
The Breakthrough: Once I stopped thinking of WMI as complex technology and started seeing it as just a database of system information, everything clicked. Don't be intimidated by the complexity. Start with simple queries and gradually explore.

Conclusion

Windows Management Instrumentation is one of the most underrated and misunderstood features of Windows. It's not magic - it's simply a built-in database that knows everything about your computer and can answer your questions if you know how to ask them.
